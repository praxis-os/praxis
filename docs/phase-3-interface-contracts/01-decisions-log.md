## D31 — `EventType` Go representation: typed string

**Status:** decided
**Summary:** `EventType` is `type EventType string` with named constants.

**Decision.** `EventType` is a typed string:

```go
type EventType string

const (
    EventTypeInvocationStarted       EventType = "invocation.started"
    EventTypeInitialized             EventType = "invocation.initialized"
    EventTypePreHookStarted          EventType = "prehook.started"
    EventTypePreHookCompleted        EventType = "prehook.completed"
    EventTypeLLMCallStarted          EventType = "llmcall.started"
    EventTypeLLMCallCompleted        EventType = "llmcall.completed"
    EventTypeToolDecisionStarted     EventType = "tooldecision.started"
    EventTypeToolCallStarted         EventType = "toolcall.started"
    EventTypeToolCallCompleted       EventType = "toolcall.completed"
    EventTypePostToolFilterStarted   EventType = "posttoolfilter.started"
    EventTypePostToolFilterCompleted EventType = "posttoolfilter.completed"
    EventTypeLLMContinuationStarted  EventType = "llmcontinuation.started"
    EventTypePostHookStarted         EventType = "posthook.started"
    EventTypePostHookCompleted       EventType = "posthook.completed"
    EventTypeInvocationCompleted     EventType = "invocation.completed"
    EventTypeInvocationFailed        EventType = "invocation.failed"
    EventTypeInvocationCancelled     EventType = "invocation.cancelled"
    EventTypeBudgetExceeded          EventType = "budget.exceeded"
    EventTypeApprovalRequired        EventType = "approval.required"
)
```

**Rationale.** The decisive factor is consumer ergonomics at the channel-drain
boundary. Consumers draining a `<-chan InvocationEvent` will switch on
`EventType` in log lines, JSON serialization, and SSE message-type headers.
A typed string marshals to JSON without a lookup table, is grep-friendly in
production logs, and produces readable `fmt.Stringer` output at zero extra cost
(`string(e.Type)` is the stringer). The hot-path performance argument for
`uint8` is real but weak: `InvocationEvent` carries a `time.Time`, a string
invocation ID, and a `BudgetSnapshot` — the type field is not the allocation
bottleneck. The 19 event constants are allocated once at program start. No
`iota` is needed; the string values are self-documenting.

**Alternatives considered.** (a) `type EventType uint8` with iota — rejected.
JSON encoding requires a map or a `MarshalJSON` method; grep/log readability
requires a `String()` method. Neither is difficult, but both are surface area
that must be maintained through the v1.0 freeze. The marginal performance gain
does not justify the ergonomic cost. (b) `type EventType int` with a `String()`
stringer generated by `stringer` tool — rejected. Code generation in the hot
path is a dependency and a `go generate` workflow burden for library consumers
who fork or vendor the package.

## D32 — `InvocationEvent` package placement: `praxis` root package

**Status:** decided
**Summary:** `InvocationEvent` and `EventType` live in the `praxis` root package.

**Decision.** `InvocationEvent`, `EventType`, and all 19 event-type constants
are exported from the `praxis` root package. The `AgentOrchestrator` facade
lives in the `praxis` root package as well (not a sub-package).

**Repository layout implication.** The seed §7 shows `orchestrator/` as a
sub-package; this decision amends the layout. The root package `praxis` is the
single import path a consumer needs for the primary usage pattern:

```go
import "MODULE_PATH_TBD"

orch := praxis.NewOrchestrator(provider)
events := orch.InvokeStream(ctx, req)
for e := range events {
    switch e.Type {
    case praxis.EventTypeToolCallStarted:
        // ...
    }
}
```

No sub-package import is required to drain the stream. Sub-packages
(`llm`, `tools`, `hooks`, `budget`, `errors`, `telemetry`, `credentials`,
`identity`, `state`) continue to exist for their respective type definitions and
default implementations; they are imported when the caller wires up
non-default dependencies.

**Import cycle analysis.** Moving `AgentOrchestrator` and `InvocationEvent` to
the root package resolves the import cycle risk flagged in the `00-plan.md`
key questions: `tools.Invoker` receives a `tools.InvocationContext` (defined in
the `tools` package), and `budget.Guard` receives a `budget.BudgetSnapshot`
(defined in the `budget` package). Neither sub-package needs to import `praxis`
root for these operations. The root package imports the sub-packages
(`llm`, `tools`, `hooks`, `budget`, `errors`, `telemetry`, `credentials`,
`identity`, `state`) to wire the constructor and define `InvocationRequest` and
`InvocationResult`. Sub-packages do not import the root. No cycle.

**Alternatives considered.** (a) `orchestrator/` sub-package for
`AgentOrchestrator` and `InvocationEvent` — rejected. Consumers must import
both the root and the sub-package for the primary use case. Every consumer file
that drains a stream needs `orchestrator.EventTypeToolCallStarted` — an
unnecessarily verbose import path for a primary type. (b) Put `EventType` in
`telemetry/` — rejected. `InvocationEvent` is the fundamental streaming
transport type; it belongs at the same level as the orchestrator, not buried in
the observability sub-package.

## D33 — `hooks.PolicyHook.Decision` type shape: struct with `Verdict` + opaque metadata

**Status:** decided
**Summary:** `Decision` is a struct with a `Verdict` typed string and an opaque
`Metadata map[string]any` field. Constructors `Allow()`, `Deny()`,
`RequireApproval()` are shipped as package-level functions.

**Decision.** See `04-hooks-and-filters.md` for the full type definitions. The
shape is:

```go
type Verdict string

const (
    VerdictAllow           Verdict = "allow"
    VerdictDeny            Verdict = "deny"
    VerdictRequireApproval Verdict = "require_approval"
    VerdictLog             Verdict = "log"   // pass-through with audit note
)

type Decision struct {
    Verdict  Verdict
    Metadata map[string]any // opaque; framework never inspects keys
    Reason   string         // human-readable; may be empty
}
```

D17 requires three things: value-variant (the loop must detect `RequireApproval`
from the `Decision` value alone, not from an error), opaque metadata forwarding,
and minimum variants `Allow`/`Deny`/`RequireApproval`. All three are satisfied.
`VerdictLog` is added per D17's "permitted if Phase 3 finds it useful" clause:
it allows a hook to note the call in an audit trail without blocking it, which
is a common pattern in policy enforcement libraries and reduces boilerplate.

The `Metadata` field is `map[string]any` rather than `[]byte` or a typed
interface to allow hooks to return rich, structured approval metadata (approval
rule IDs, required approver groups, reason codes) without requiring the caller
to serialize/deserialize. The framework forwards the map opaquely into the
`ApprovalRequiredError` and the `EventTypeApprovalRequired` payload without
unmarshaling or inspecting any key.

**Alternatives considered.** (a) Sealed variant using a private interface —
rejected. In Go, a sealed interface (unexported method) prevents external
implementations, which conflicts with our design principle of accepting
interface implementations from callers. A hook returning a `Decision` via an
opaque interface value would force callers to use only the framework-provided
constructors, which is the right behavior, but it means the `Decision` cannot
be value-type-switched cleanly. The struct approach achieves the same routing
guarantees with a public type. (b) `(Verdict, []byte)` tuple — rejected. JSON
blobs for metadata are awkward in Go and force a serde round-trip for metadata
the framework never needs to read. (c) Separate `ApprovalDecision` type for the
`RequireApproval` variant — rejected. Keeping one `Decision` type simplifies the
`PolicyHook` return signature and avoids a union type in the return position.

## D34 — `budget.Guard` method surface: snapshot-based `Check` + dimension-additive recording

**Status:** decided
**Summary:** `Guard` exposes two concerns: `RecordXxx` methods for dimension
accumulation and a `Check` method that reads accumulated state against limits.
The loop calls `RecordXxx` at the appropriate states and `Check` before each
budget-gated transition.

**Decision.** The `budget.Guard` interface is:

```go
type Guard interface {
    RecordTokens(ctx context.Context, inputTokens, outputTokens int64) error
    RecordToolCall(ctx context.Context) error
    RecordCost(ctx context.Context, microdollars int64) error
    Check(ctx context.Context) (BudgetSnapshot, error)
    Snapshot(ctx context.Context) BudgetSnapshot
}
```

`RecordXxx` methods accumulate consumed resources. `Check` returns the current
snapshot and a non-nil `BudgetExceededError` if any configured dimension has
been breached. `Snapshot` returns the current snapshot without performing a
breach check — used by the loop to populate `InvocationEvent.BudgetSnapshot`
on non-terminal events.

CP3 requires that the same `Guard` instance can be shared across nested
orchestrators. The dimension-accumulation model satisfies CP3 directly: both
the outer and inner orchestrators call `RecordXxx` on the shared instance, and
`Check` reports against the combined total. The framework never assumes a
one-to-one mapping between a `Guard` and an invocation.

Wall-clock duration is not a `RecordXxx` parameter; it is computed inside the
`Guard` implementation from the invocation start time, which is set via the
constructor of the default `BudgetGuard` implementation. This keeps the
interface free of `time.Time` parameters that would complicate shared-instance
semantics.

**Alternatives considered.** (a) Single `Check(snapshot BudgetSnapshot) error`
method with the loop computing the snapshot externally — rejected. The loop does
not know the wall-clock start time or the accumulated state if the Guard is
shared; the Guard must own that state. (b) Single `Record(snapshot
BudgetSnapshot) (BudgetSnapshot, error)` — rejected. Passing a pre-constructed
snapshot as input conflates reading state (the caller's job) with owning state
(the Guard's job) and makes the shared-instance semantic ambiguous. (c) Separate
`WallClockGuard`, `TokenGuard`, `CostGuard`, `ToolCallGuard` interfaces composed
in a `Guard` umbrella — rejected. Over-engineered for v1.0. The four dimensions
are always deployed together; decomposition provides no practical benefit.

## D35 — `BudgetSnapshot` struct shape

**Status:** decided
**Summary:** Value-copyable struct with four consumed-resource fields and an
`ExceededDimension` sentinel.

**Decision.** See `05-budget-interfaces.md` for full type definitions. Shape:

```go
type BudgetDimension string

const (
    BudgetDimensionWallClock  BudgetDimension = "wall_clock"
    BudgetDimensionTokens     BudgetDimension = "tokens"
    BudgetDimensionToolCalls  BudgetDimension = "tool_calls"
    BudgetDimensionCost       BudgetDimension = "cost"
)

type BudgetSnapshot struct {
    ElapsedWallClock  time.Duration
    InputTokensUsed   int64
    OutputTokensUsed  int64
    ToolCallsUsed     int64
    CostMicrodollars  int64
    ExceededDimension BudgetDimension // empty string when no breach
}
```

Zero value is safe: all numeric fields are zero, `ExceededDimension` is empty
string (no breach). The struct is value-copyable (no pointers) so it can be
passed through channels and stored in event history without aliasing concerns.
`ElapsedWallClock` is a `time.Duration` (int64 nanoseconds internally); no
`time.Time` pointers needed. `CostMicrodollars` uses int64 micro-dollars
(millionths of a dollar), matching the budget dimension established in seed §5.

**Alternatives considered.** (a) Separate `LimitSnapshot` for configured
limits alongside consumed values — rejected. Limits are configuration, not
runtime state; they belong on the `Guard` constructor, not in a runtime
snapshot. Embedding limits in the snapshot would require the Guard to expose
its configuration to every event receiver. (b) `map[BudgetDimension]int64` —
rejected. The fixed four dimensions do not benefit from a map; the struct is
more efficient and avoids allocations on the event path.

## D36 — `tools.InvocationContext` field set

**Status:** decided
**Summary:** Struct with invocation ID, read-only budget snapshot, span context,
outer signed identity token, and invocation metadata. No tool name (D06).

**Decision.** See `06-tools-and-invocation-context.md` for full definitions.
Shape:

```go
// InvocationContext carries framework-maintained state for a single
// invocation. It is passed to tools.Invoker.Invoke and is read-only:
// the invoker must not mutate any field.
//
// InvocationContext is safe for concurrent use — multiple Invoke calls
// may share a read-only InvocationContext under parallel tool dispatch (D24).
type InvocationContext struct {
    // InvocationID is the unique identifier for this invocation (D49).
    InvocationID string

    // Budget is the current budget snapshot at the time of this tool call.
    // It is a read-only value; the Guard is not exposed to the invoker.
    Budget budget.BudgetSnapshot

    // SpanContext is the OTel span context for the current invocation,
    // allowing the invoker to create child spans for its own operations.
    // May be zero if telemetry is not configured.
    SpanContext trace.SpanContext

    // SignedIdentity is the JWT produced by identity.Signer for this call,
    // or an empty string if identity signing is not configured.
    // The invoker may forward this to the downstream tool as a bearer token.
    SignedIdentity string

    // Metadata is a caller-controlled key-value map attached to the
    // InvocationRequest. The framework forwards it to the invoker unchanged.
    // The framework never inspects the keys or values.
    Metadata map[string]string
}
```

`InvocationContext` is a struct, not an interface, because it is a transparent
data container. Making it an interface would require callers to write test
doubles that implement each field accessor — unnecessary complexity.

`SpanContext` uses `go.opentelemetry.io/otel/trace.SpanContext` (a value type,
not a pointer) so the context is copyable without heap allocation. If no tracer
is configured, `SpanContext{}` is the zero value and the invoker can skip span
creation safely.

`Metadata` is `map[string]string` (not `map[string]any`) to stay string-typed
and JSON-serializable without a type assertion. Callers with richer metadata
needs can serialize to JSON strings.

**What `InvocationContext` does not carry:** the tool name (D06, lives on
`ToolCall`), the `budget.Guard` instance (invokers must not record budget
against the outer guard), the `llm.Provider` instance, any hook or filter.

**Alternatives considered.** (a) Interface with accessor methods — rejected;
adds indirection with no benefit for a pure data container. (b) Embed the full
`context.Context` in `InvocationContext` — rejected; the Go `context.Context`
is already threaded through the `Invoke` method's first parameter; duplicating
it in the struct is confusing.

## D37 — `AgentOrchestrator` constructor and functional options

**Status:** decided
**Summary:** `NewOrchestrator(provider llm.Provider, opts ...Option) *Orchestrator`
with a complete functional option inventory.

**Decision.** See `02-orchestrator-api.md` for the full option inventory. The
constructor signature is:

```go
func NewOrchestrator(provider llm.Provider, opts ...Option) *Orchestrator
```

`Orchestrator` is a concrete struct (not an interface named
`AgentOrchestrator`). An `Orchestrator` is safe for concurrent use. Calling
`Invoke` or `InvokeStream` on the same `Orchestrator` from multiple goroutines
is supported.

Options follow the functional-options pattern with a private `options` struct
and an exported `Option` type. All dependency injections are options:

```go
type Option func(*options)

func WithToolInvoker(invoker tools.Invoker) Option
func WithPolicyHook(hook hooks.PolicyHook) Option
func WithPreLLMFilter(filter hooks.PreLLMFilter) Option
func WithPostToolFilter(filter hooks.PostToolFilter) Option
func WithBudgetGuard(guard budget.Guard) Option
func WithPriceProvider(pp budget.PriceProvider) Option
func WithLifecycleEventEmitter(emitter telemetry.LifecycleEventEmitter) Option
func WithAttributeEnricher(enricher telemetry.AttributeEnricher) Option
func WithCredentialResolver(resolver credentials.Resolver) Option
func WithIdentitySigner(signer identity.Signer) Option
func WithClassifier(classifier errors.Classifier) Option
func WithInvocationMetadata(metadata map[string]string) Option
```

All options default to their respective null implementations if not provided.
The zero-wiring path (`NewOrchestrator(provider)`) constructs a fully functional
orchestrator per D12.

**Rationale.** Functional options are the idiomatic Go pattern for constructors
with optional dependencies (> 3 parameters). They are composable (callers can
define their own `Option` slices), testable (each option is independently
testable), and extensible within a release (new options can be added without
changing the constructor signature). The requirement from D12 that
`llm.Provider` is the single required dependency is enforced by making it the
positional first argument.

**Alternatives considered.** (a) `Config` struct passed by value — rejected.
Config structs have zero-value ambiguity (is a nil `PolicyHook` field intentional
or was the caller confused?). Functional options make intent explicit.
(b) Builder pattern with chained methods — rejected. Go builders require a
distinct builder type and a terminal `Build()` call; functional options achieve
the same result with less boilerplate.

## D38 — `InvocationRequest` and `InvocationResult` shapes

**Status:** decided
**Summary:** `InvocationRequest` carries messages, model, tools, and budget
config. `InvocationResult` carries the response, terminal state, budget
snapshot, and collected events.

**Decision.** See `02-orchestrator-api.md` for full definitions. Shapes:

```go
type InvocationRequest struct {
    // Messages is the conversation history, in provider-agnostic format.
    // Must contain at least one message; the last message is typically a user turn.
    Messages []llm.Message

    // Model is the model identifier to use for this invocation.
    // Format is provider-specific (e.g., "claude-opus-4-5", "gpt-4o").
    Model string

    // Tools is the list of tools available to the LLM for this invocation.
    // If empty, the LLM is invoked without tool use.
    Tools []llm.ToolDefinition

    // BudgetConfig configures per-invocation budget limits.
    // Zero value means no limits are enforced (the Guard still records).
    BudgetConfig budget.Config

    // SystemPrompt is an optional system prompt prepended to the conversation.
    SystemPrompt string

    // Metadata is an opaque key-value map forwarded to tools.InvocationContext.
    // The framework never inspects the keys or values.
    Metadata map[string]string

    // MaxTurns limits the number of LLM-tool cycles.
    // Zero means no limit (cycle count is enforced by budget.Guard tool-call dimension).
    MaxTurns int
}

type InvocationResult struct {
    // InvocationID is the unique ID assigned by the orchestrator.
    InvocationID string

    // FinalState is the terminal state reached (Completed, Failed, Cancelled,
    // BudgetExceeded, or ApprovalRequired).
    FinalState state.State

    // Response is the LLM's final response, in provider-agnostic format.
    // Nil if the invocation did not reach a Completed state.
    Response *llm.Message

    // BudgetSnapshot is the budget state at invocation termination.
    BudgetSnapshot budget.BudgetSnapshot

    // Events is the ordered list of all InvocationEvents emitted during the
    // invocation. On the Invoke path, this is the full history. On the
    // InvokeStream path, callers drain the channel directly and do not use
    // this field (it will be nil).
    Events []InvocationEvent
}
```

**Rationale.** `InvocationRequest` is a struct rather than a functional options
list because it is a data transfer object, not a configuration accumulator. It
is short (7 fields), all fields have clear zero-value semantics, and callers
typically construct it inline. `InvocationResult.Events` is nil on the
`InvokeStream` path because the caller consumes events from the channel
directly; populating this field from the stream would require buffering all
events in memory, which is a caller choice, not a framework obligation.

**Alternatives considered.** (a) `InvocationResponse` instead of
`InvocationResult` — naming preference; `Result` aligns with Go convention
(functions return `(Result, error)`). (b) Return `InvocationResult` on
`InvokeStream` — rejected. `InvokeStream` returns a channel; attaching a
`Result` would require an additional blocking call after the channel drains,
complicating the streaming API.

## D39 — `ApprovalRequiredError` and conversation snapshot shape

**Status:** decided
**Summary:** `ApprovalRequiredError` contains an `ApprovalSnapshot` with the
full message history, the original request sufficient for re-invocation, the
budget snapshot at approval time, and the opaque approval metadata from the
`Decision`.

**Decision.** See `07-errors-and-classifier.md` for full definitions. Shape:

```go
type ApprovalSnapshot struct {
    // Messages is the full conversation history up to the approval point,
    // in provider-agnostic format. Includes all tool calls and tool results.
    Messages []llm.Message

    // OriginalRequest is the InvocationRequest that produced this approval
    // checkpoint, sufficient for the caller to re-invoke after approval.
    OriginalRequest InvocationRequest

    // BudgetAtApproval is the budget snapshot at the moment of the
    // ApprovalRequired transition. Callers may use this to configure a
    // budget offset for the resuming invocation.
    BudgetAtApproval budget.BudgetSnapshot

    // ApprovalMetadata is the opaque metadata from the Decision.Metadata
    // field of the PolicyHook that triggered this approval requirement.
    // The framework does not inspect it; callers define its schema.
    ApprovalMetadata map[string]any
}

type ApprovalRequiredError struct {
    Snapshot ApprovalSnapshot
    // wrapped error is nil; ApprovalRequired is not a failure
    err error
}

func (e *ApprovalRequiredError) Kind() errors.ErrorKind       { return errors.ErrorKindApprovalRequired }
func (e *ApprovalRequiredError) HTTPStatusCode() int          { return 202 } // Accepted, pending approval
func (e *ApprovalRequiredError) Unwrap() error                { return e.err }
func (e *ApprovalRequiredError) Error() string                { ... }
```

HTTP status 202 (Accepted) is a deliberate choice: the request was accepted and
is in a pending state, not failed. This gives callers an unambiguous HTTP hint
that distinguishes "approval pending" from "internal error" (500) or "denied"
(403).

**D50 resolution.** Resume after `ApprovalRequired` is a fresh `Invoke` call
with a new `InvocationRequest` containing the snapshot's `Messages` extended by
the human approval result. D07's position is confirmed without amendment.

**Alternatives considered.** (a) HTTP 403 Forbidden for `ApprovalRequired` —
rejected. 403 implies the request is denied permanently; an approval-pending
request may succeed with approval. (b) A dedicated `Resume(snapshot) (Result,
error)` method on `Orchestrator` — rejected. This would require the
orchestrator to understand resume semantics internally, adding state to what
is otherwise a stateless facade. A fresh `Invoke` with an extended message
history is idiomatic and requires no framework changes.

## D40 — `tools.ToolResult` shape with typed-error embedding (CP5)

**Status:** decided
**Summary:** `ToolResult` carries a `Status`, a `Content` string (provider-agnostic
tool output), and an optional `Err` field of type `error` that preserves
`TypedError` provenance for nested invocation propagation.

**Decision.** See `06-tools-and-invocation-context.md` for full definitions.
Shape:

```go
type ToolStatus string

const (
    ToolStatusSuccess          ToolStatus = "success"
    ToolStatusDenied           ToolStatus = "denied"
    ToolStatusNotImplemented   ToolStatus = "not_implemented"
    ToolStatusError            ToolStatus = "error"
)

type ToolResult struct {
    // Status indicates the outcome of the tool invocation.
    Status ToolStatus

    // Content is the tool's output in the provider-agnostic format.
    // May be structured JSON or plain text, at the invoker's discretion.
    // Empty string is valid for Denied and NotImplemented statuses.
    Content string

    // Err carries the underlying error for ToolStatusError, or a propagated
    // TypedError from a nested invocation for CP5. The outer orchestrator's
    // Classifier inspects Err via errors.As before reclassifying.
    // For ToolStatusSuccess, ToolStatusDenied, and ToolStatusNotImplemented,
    // Err must be nil.
    Err error

    // CallID is echoed from the ToolCall.CallID that produced this result.
    CallID string
}
```

CP5 requires two things: (1) `ToolResult` carries enough structure to embed a
`TypedError`, and (2) the outer orchestrator's classifier preserves propagated
typed errors rather than collapsing them into `ToolError{SchemaViolation}`. The
`Err error` field satisfies (1). The classifier precedence rule (classify using
`errors.As(err, &typedErr)` before applying the default `ToolError`
classification) satisfies (2); this is documented in `07-errors-and-classifier.md`.

Using `error` (rather than `errors.TypedError`) for the `Err` field preserves
the "accept interfaces, return concrete types" idiom and avoids a circular import
(`tools` importing `errors` for the interface is fine; `errors` importing
`tools` would not be).

**Alternatives considered.** (a) `TypedErr errors.TypedError` typed field —
rejected. `errors.TypedError` is an interface; using it as a struct field type
requires `nil` comparisons instead of `== nil`, and the `errors.As` approach is
idiomatic. (b) A separate `PropagatedError` wrapper type — rejected. Adds a
type that callers must unwrap. The plain `error` field with classifier precedence
rules is simpler and more idiomatic.

## D41 — `llm.Provider` method surface and agnostic type shapes

**Status:** decided
**Summary:** Five methods: `Complete`, `Stream`, `Name`, `SupportsParallelToolCalls`,
`Capabilities`. `Stream` returns a channel of `LLMStreamChunk`. Agnostic types
defined in the `llm` package.

**Decision.** See `03-llm-provider.md` for full type definitions. Interface:

```go
type Provider interface {
    Complete(ctx context.Context, req LLMRequest) (LLMResponse, error)
    Stream(ctx context.Context, req LLMRequest) (<-chan LLMStreamChunk, error)
    Name() string
    SupportsParallelToolCalls() bool
    Capabilities() Capabilities
}
```

`Stream` returns a `(<-chan LLMStreamChunk, error)` rather than a
`(LLMResponse, error)` to give the orchestrator the ability to forward
streaming tokens to the `InvokeStream` channel. The error return on
`Stream` is for setup failures (auth error, network error before first
token); once the channel is open, errors are embedded in chunks.
`Complete` is the non-streaming path; adapters that do not natively
support streaming must implement `Stream` by buffering the full response.

**Alternatives considered.** (a) `Complete` only, with a streaming option
field on `LLMRequest` — rejected. Conflates two fundamentally different
response patterns (buffered vs. streamed) in a single return type; makes
the adapter contract ambiguous. (b) `Stream` returning `iter.Seq` (Go 1.23
range-over-function) — considered. The range-over-func pattern is cleaner
for sequential consumption, but it is a new Go 1.23 pattern that adapter
authors and mock authors may not be familiar with. A channel is more
familiar, is idiomatic for the buffered-16 model, and is composable with
the existing `select` patterns. Reconsidered post-v1.0 as `ProviderV2`.

## D42 — Filter chain return types

**Status:** decided
**Summary:** `PreLLMFilter.Filter` returns `([]llm.Message, []FilterDecision, error)`.
`PostToolFilter.Filter` returns `(tools.ToolResult, []FilterDecision, error)`.

**Decision.** See `04-hooks-and-filters.md` for full definitions. The
`FilterDecision` type:

```go
type FilterAction string

const (
    FilterActionPass   FilterAction = "pass"
    FilterActionRedact FilterAction = "redact"
    FilterActionLog    FilterAction = "log"
    FilterActionBlock  FilterAction = "block"
)

type FilterDecision struct {
    Action FilterAction
    // Field is the path within the filtered content that was acted on.
    // Empty string means the decision applies to the whole input.
    Field  string
    Reason string
}
```

The `decisions []FilterDecision` return allows a single filter invocation to
report multiple decisions on different parts of the content (e.g., "redacted
field X, logged field Y"). This enables Phase 4's error-to-event mapping to
identify `EventTypePIIRedacted` and `EventTypePromptInjectionSuspected`
without adding those concerns to the filter interface itself. A `FilterAction`
of `Block` in any returned decision causes the loop to route the invocation to
`Failed`.

**Alternatives considered.** (a) Single `FilterDecision` return (not a slice) —
rejected. A filter that both redacts and logs cannot express both actions in one
value. (b) Return `(filtered, error)` with the decision embedded in a special
error type — rejected. Conflates operational failures (the filter panicked or
timed out) with policy decisions (the filter chose to redact). These are
distinct.

## D43 — `state.State` Go type

**Status:** decided
**Summary:** `type State uint8` with `iota` constants in the order non-terminals
first, terminals last. `IsTerminal()` uses a single range comparison. Adjacency
table exported read-only.

**Decision.** See `10-state-types.md` for full definitions. Shape:

```go
type State uint8

const (
    Created          State = iota // 0
    Initializing                  // 1
    PreHook                       // 2
    LLMCall                       // 3
    ToolDecision                  // 4
    ToolCall                      // 5
    PostToolFilter                // 6
    LLMContinuation               // 7
    PostHook                      // 8
    Completed                     // 9  -- first terminal
    Failed                        // 10
    Cancelled                     // 11
    BudgetExceeded                // 12
    ApprovalRequired              // 13
    stateCount                    // 14 -- sentinel, unexported
)

func (s State) IsTerminal() bool { return s >= Completed }
func (s State) String() string   { ... }
```

The adjacency table is exposed as a read-only function, not a map variable,
to prevent accidental mutation:

```go
// Transitions returns the set of states reachable from s in one step.
// Returns nil for terminal states (no outgoing edges).
// This function is the authoritative source for property-based tests.
func Transitions(s State) []State
```

**Rationale.** The `IsTerminal()` range comparison (`s >= Completed`) is the
exactly correct predicate if and only if terminals are grouped at the end of the
iota sequence — which they are by the D15 ordering. This avoids a set-membership
check and is branchless on most architectures.

**Alternatives considered.** (a) `type State string` — rejected. Transitions
and terminal checks become string comparisons; the iota ordering trick is lost.
(b) Expose the adjacency table as `map[State][]State` — rejected. A package-level
variable is mutable at runtime (even if documented as read-only, nothing enforces
it). A function returning a fresh slice is immutable by construction.

## D44 — `errors.Classifier` method signature

**Status:** decided
**Summary:** `Classify(error) errors.TypedError`. Returns the input wrapped as a
`TypedError`; never returns nil.

**Decision.** See `07-errors-and-classifier.md` for full definitions. Interface:

```go
type Classifier interface {
    Classify(err error) TypedError
}
```

The classifier always returns a non-nil `TypedError`. If the input error is
already a `TypedError` (via `errors.As`), the classifier returns it unchanged
(identity rule). If the input is a plain `error`, the classifier wraps it in a
`SystemError` or `TransientLLMError` depending on classification heuristics.

CP5 classifier precedence: the default classifier must check for a propagated
`TypedError` (via `errors.As(err, &typed)`) before applying any heuristic
reclassification. This ensures that a `BudgetExceededError` surfaced through a
`ToolResult.Err` is returned as-is, not collapsed into `ToolError`.

**Alternatives considered.** (a) `Classify(error) (ErrorKind, error)` — rejected.
Returning two values is awkward when the caller needs to inspect the full
`TypedError` (including `HTTPStatusCode()` and `Unwrap()`). A single
`TypedError` return carries all needed fields. (b) `Classify(error) (TypedError,
bool)` — rejected. The bool is always true (the classifier always produces a
result); it adds noise with no information. (c) `Classify(error) (TypedError,
error)` — rejected. A classifier that can itself fail is a design smell; errors
in classification are logged and fall back to `SystemError`, not propagated as
errors.

## D45 — `credentials.Resolver` and `Credential` shapes

**Status:** decided
**Summary:** `Resolver` has a single `Fetch(ctx, ref) (Credential, error)` method.
`Credential` is an interface with `Value() []byte` and `Close() error`.

**Decision.** See `09-credentials-and-identity.md` for full definitions.

```go
type Resolver interface {
    Fetch(ctx context.Context, ref CredentialRef) (Credential, error)
}

type CredentialRef struct {
    // Name is the logical name of the credential (e.g., "stripe-api-key").
    // The Resolver interprets this name; the framework never inspects it.
    Name string

    // Scope is an optional scope hint (e.g., "read", "write").
    // May be empty; interpretation is Resolver-specific.
    Scope string
}

type Credential interface {
    // Value returns the secret material. Callers must not retain this slice
    // after calling Close(). The framework zeroes the slice on Close().
    Value() []byte

    // Close zeroes secret material in memory and releases any associated
    // resources. Must be called exactly once. Safe to call multiple times
    // (idempotent). The framework calls Close() in a deferred call after
    // each tool invocation.
    Close() error
}
```

`CredentialRef` is a struct (not a string) to allow extension without interface
changes. The `Scope` field is included now because credential systems universally
support scoped tokens, and adding it post-freeze would require a `CredentialRefV2`.

**Alternatives considered.** (a) `Fetch(ctx, name string) ([]byte, error)` with
no `Credential` interface — rejected. The caller cannot control the lifecycle of
the secret material; the bytes would need to be zeroed by the framework with no
ownership guarantee. The `Credential` interface's `Close()` method makes the
lifecycle contract explicit and testable. (b) `Credential` as a concrete struct
with `Value []byte` — rejected. A concrete struct exposes the raw bytes without
a lifecycle contract; callers and mocks cannot control when zeroing happens.

## D46 — `identity.Signer` method surface (stable-v0.x-candidate)

**Status:** decided (stable-v0.x-candidate; promotion to frozen-v1.0 deferred to Phase 5)
**Summary:** One method: `Sign(ctx, invocationID, toolName string) (string, error)`.

**Decision.** See `09-credentials-and-identity.md` for full definitions.

```go
type Signer interface {
    Sign(ctx context.Context, invocationID string, toolName string) (string, error)
}
```

`Sign` returns a short-lived Ed25519 JWT as a string. The empty string is a
valid return for `NullSigner`. Parameters are: the invocation ID (for the JWT
subject or `jti` claim), and the tool name (for a claim identifying the
operation). The full JWT claim set is Phase 5's jurisdiction; this interface
carries only the inputs required for signing.

The seed §5 shape (`Sign(ctx, invocation, toolName) (string, error)`) used an
`invocation` struct; this decision replaces it with `invocationID string` to
keep the `identity` package free of `praxis`-root-package imports (avoiding an
import cycle where `identity` imports `praxis` for an `Invocation` type).

**Alternatives considered.** (a) `Sign(ctx context.Context, req SignRequest)
(string, error)` with a `SignRequest` struct — retained as a future shape for
`ProviderV2` in case Phase 5 needs to add claim parameters without breaking the
interface. For v0.x, the flat parameter list is simpler. (b) Returning
`*jwt.Token` — rejected. Coupling the interface to a specific JWT library.

## D47 — `budget.PriceProvider` final signature: promoted to `frozen-v1.0`

**Status:** decided; `budget.PriceProvider` is hereby promoted to `frozen-v1.0`
**Summary:** `PriceForToken(provider, model string, direction TokenDirection) (int64, error)`.
Per-invocation snapshot semantics (D08) make this interface mechanically stable.

**Decision.** See `05-budget-interfaces.md` for full definitions.

```go
type TokenDirection string

const (
    TokenDirectionInput  TokenDirection = "input"
    TokenDirectionOutput TokenDirection = "output"
)

type PriceProvider interface {
    PriceForToken(ctx context.Context, provider, model string, direction TokenDirection) (int64, error)
}
```

Return value is the price in micro-dollars per token (millionths of a dollar).
Returning `int64` avoids floating-point precision issues for cost accumulation.
The `ctx` parameter is included per the Go idiom checklist (any method that
may perform I/O — a remote price lookup, a config file read — takes context).

The D08 per-invocation snapshot means this method is called at most once per
`(provider, model, direction)` tuple per invocation start. Hot-path performance
of `PriceForToken` is not a concern.

**Promotion to `frozen-v1.0`** takes effect at Phase 3 sign-off, as scheduled
in `docs/phase-1-api-scope/04-v1-freeze-surface.md` §Promotion schedule.

**Alternatives considered.** (a) `Price(req PriceRequest) (int64, error)` — a
`PriceRequest` struct would allow future extension without breaking the
interface, but D13's preference for small interfaces and D08's snapshot
semantics make it unlikely that new parameters will be needed. The flat
signature is chosen for v1.0; a `PriceProviderV2` embedding `PriceProvider` is
the extension path if needed. (b) `float64` return — rejected. Floating-point
accumulation of micro-dollar costs produces rounding errors; `int64` is exact
for the micro-dollar granularity.

## D48 — `AgentOrchestrator.Close()`: excluded from v1.0

**Status:** decided
**Summary:** `*Orchestrator` has no `Close()` method. The orchestrator does not
own long-lived background goroutines or connection pools.

**Decision.** `*Orchestrator` does not implement `io.Closer`. No `Close()` method
is added to the public surface.

**Rationale.** Phase 2 `05-concurrency-model.md` §5 established that the
orchestrator spawns no background goroutines outside of individual
`Invoke`/`InvokeStream` calls. Each invocation's goroutine terminates when the
invocation reaches a terminal state. The orchestrator is a stateless facade
that holds references to injected dependencies; it owns no resources that need
explicit cleanup. Injected dependencies (`llm.Provider`, `credentials.Resolver`,
etc.) are owned by the caller, who is responsible for their lifecycle.

Adding `Close()` to the v1.0 surface would commit to a lifecycle contract the
orchestrator does not need, and would mislead callers into thinking the
orchestrator holds resources that require cleanup.

**Forward record.** If a v2 orchestrator introduces a connection pool or
background maintenance goroutine, `Close()` can be added to a
`OrchestratorV2` interface embedding `*Orchestrator`. This is not a v1.0
concern.

## D49 — Invocation ID generation: UUIDv7

**Status:** decided
**Summary:** Invocation IDs are UUIDv7 strings (RFC 9562), generated by the
orchestrator at invocation start. Lexicographically sortable by creation time.

**Decision.** The orchestrator generates a UUIDv7 at the start of each
invocation using a stdlib-compatible generator. The invocation ID is exposed as
a `string` in `InvocationResult.InvocationID`, `InvocationContext.InvocationID`,
and all `InvocationEvent.InvocationID` fields.

UUIDv7 is chosen over UUIDv4 for one reason: lexicographic sort order matches
creation time order, which is valuable for log and database queries that filter
by invocation order. UUIDv7 encodes a millisecond-precision Unix timestamp in
the most-significant bits, making `ORDER BY invocation_id` equivalent to
`ORDER BY created_at` without a secondary index.

The representation is the standard hyphenated UUID string
(`xxxxxxxx-xxxx-7xxx-xxxx-xxxxxxxxxxxx`). No custom types; `string` is
sufficient.

**Dependency note.** `encoding/binary` and `crypto/rand` from stdlib are
sufficient to generate UUIDv7. No external UUID library is required, consistent
with the project's preference for minimal dependencies. The implementation lives
in `internal/` and is not part of the public API.

**Alternatives considered.** (a) UUIDv4 — rejected. No time-order property;
log queries require a separate timestamp field for ordering. (b) ULID — rejected.
ULID is a non-standard format with a custom alphabet; UUIDv7 is an IETF RFC
standard and is becoming the dominant choice for new databases (PostgreSQL 17,
MySQL 8.4). (c) Caller-provided invocation ID via `InvocationRequest` — rejected
as the primary generation strategy. The orchestrator must guarantee uniqueness;
caller-provided IDs could collide. Callers who need to correlate a praxis
invocation with their own request ID should use `Metadata` and
`AttributeEnricher`.

## D50 — Resume after `ApprovalRequired`: confirmed as fresh `Invoke`

**Status:** decided (confirms D07, no amendment)
**Summary:** Resume is a fresh `Invoke` call with a new `InvocationRequest`
derived from the `ApprovalSnapshot`. No `Resume` method is added to
`*Orchestrator`.

**Decision.** D07's position is confirmed without amendment. When a caller
receives an `ApprovalRequiredError` (or `EventTypeApprovalRequired` on the
stream), it extracts the `ApprovalSnapshot`, appends the human approval result
as a new message to `Snapshot.Messages`, and calls `orchestrator.Invoke` with
a new `InvocationRequest` using those messages. Each resume is a new invocation
with a fresh invocation ID, fresh state machine, and independently configured
budget.

The `ApprovalSnapshot.OriginalRequest` field (D39) provides the full request
configuration (model, tools, budget config, metadata) needed to reconstruct the
`InvocationRequest` without the caller storing it separately.

**No `Resume` method added.** A dedicated `Resume(snapshot ApprovalSnapshot)
(InvocationResult, error)` method would be syntactic sugar for the pattern
above and would add a v1.0-frozen API surface with no semantic content beyond
"construct a request from the snapshot and call Invoke". The ergonomic gain is
marginal; the surface cost is real.

**Alternatives considered.** (a) `Resume(ApprovalSnapshot)` method — rejected;
see above. (b) A non-terminal `ApprovalPending` state with a resume signal —
rejected in D07 for incompatibility with cancellation, streaming, and budget
contracts; confirmed rejected here.

## D51 — Package layout resolution and OPEN-1 import cycle fix

**Status:** decided
**Supersedes:** D32 (partially — the `InvocationEvent`/`EventType` root-package
placement stands; the facade placement is amended)
**Summary:** `*Orchestrator`, `New`, and all `With*` options live in
`orchestrator/` sub-package (not the root `praxis` package). `PolicyInput` stays
in `hooks` with projection fields instead of a direct `InvocationRequest`
reference, breaking the `praxis ↔ hooks` import cycle.

**Decision.** Two interrelated structural problems surfaced during review:

1. **Facade placement contradiction.** D32 stated the `AgentOrchestrator` facade
   lives in the `praxis` root package. The go-architect analysis showed this
   creates an import cycle: `praxis` root (holding `WithPolicyHook`) → `hooks`
   (for `PolicyHook` type) → `praxis` root (for `PolicyInput.Request` of type
   `InvocationRequest`). The go-architect's approach — root holds types,
   `orchestrator/` holds behavior — avoids the cycle.

2. **OPEN-1 (`PolicyInput.Request` references `InvocationRequest`).** If
   `PolicyInput` carries `InvocationRequest` from the root package, `hooks`
   must import root. If root imports `hooks` (for the constructor's options),
   the cycle is `praxis ↔ hooks`.

**Resolution:**

- **`*Orchestrator`, `New`, and all `With*` option functions live in the
  `orchestrator/` sub-package.** Import path: `MODULE_PATH_TBD/orchestrator`.
  The root `praxis` package holds only shared types: `InvocationEvent`,
  `InvocationRequest`, `InvocationResult`, `EventType` constants, `Option` type.
  The root package does NOT import `hooks`, `tools`, or any other interface
  package. `orchestrator/` imports everything.

- **`PolicyInput` stays in `hooks` with projection fields.** Instead of carrying
  a `praxis.InvocationRequest`, `PolicyInput` carries the subset of fields that
  policy hooks actually need:

  ```go
  // PolicyInput is the context provided to PolicyHook.Evaluate.
  type PolicyInput struct {
      InvocationID string
      Model        string
      SystemPrompt string
      Messages     []llm.Message
      ToolResult   *tools.ToolResult
      LLMResponse  *llm.LLMResponse
      Metadata     map[string]string
  }
  ```

  `hooks` imports `llm` (Level 1 leaf) and `tools` (Level 2). Neither imports
  `hooks`. No cycle. The orchestrator populates `PolicyInput` from
  `InvocationRequest` before calling `Evaluate`.

**Import DAG after this decision (abridged):**

```
Level 0: state, errors, identity
Level 1: budget → errors, llm → errors, credentials → errors
Level 2: praxis → state, budget, errors, llm
         tools → budget, errors
         hooks → errors, llm, tools
         telemetry → praxis, errors
Level 3: internal/loop → everything except orchestrator
Level 4: orchestrator → praxis, llm, tools, hooks, budget, errors,
                         telemetry, credentials, identity, internal/loop
```

Zero cycles.

**Consumer smoke path after this decision:**

```go
import (
    "MODULE_PATH_TBD"
    "MODULE_PATH_TBD/orchestrator"
    "MODULE_PATH_TBD/llm/anthropic"
)

provider := anthropic.New(os.Getenv("ANTHROPIC_API_KEY"))
orch := orchestrator.New(provider)
result, err := orch.Invoke(ctx, praxis.InvocationRequest{
    Messages: []llm.Message{{Role: llm.RoleUser, Parts: []llm.MessagePart{{Text: "Hello"}}}},
})
```

Two imports for the minimal case (praxis root + orchestrator). The root import
is needed for `InvocationRequest`; `orchestrator` for `New`. This is one more
import than D32 proposed, but cycle-freedom is non-negotiable.

**Alternatives considered.** (a) Keep facade in root and move all option types
into root — rejected; root would import every interface package, reintroducing
cycles. (b) Create a separate `praxis/types` package for shared types — rejected;
unnecessary fragmentation; the root `praxis` package already fills this role.

## D52 — Seed amendments: interface→struct, event vocabulary extension

**Status:** decided
**Amends:** seed §5 (`orchestrator.AgentOrchestrator`), D18 (event count)

**Decision.** Three seed-context amendments are recorded:

### 52a. `orchestrator.AgentOrchestrator` → `*orchestrator.Orchestrator` (concrete struct)

Seed §5 describes `orchestrator.AgentOrchestrator` as an interface with a
"default impl: the bundled loop built on the eleven-state machine." D37 adopted
a concrete struct `*Orchestrator` instead. This is a deliberate departure:

- A concrete struct avoids the "God interface" anti-pattern. The orchestrator is
  the integration point, not a seam — callers never need to provide an
  alternative orchestrator implementation. Extension is via the 12 injected
  dependencies, not via replacing the orchestrator itself.
- Callers who need to mock the orchestrator in tests define their own
  narrow interface at the consumer site:

  ```go
  // In the caller's test file:
  type Invoker interface {
      Invoke(ctx context.Context, req praxis.InvocationRequest) (*praxis.InvocationResult, error)
  }
  ```

  This is idiomatic Go (accept interfaces, return concrete types) and avoids
  praxis shipping a testing interface that may not match every caller's test
  surface.

### 52b. Event vocabulary extended from 19 to 21 types

Seed §5 lists `EventPIIRedacted` and `EventPromptInjectionSuspected` as
framework-defined lifecycle event constants. D18 (Phase 2) specified 19 event
types tied to state-machine transitions. The seed's two additional events are
not transition events — they are content-analysis events emitted by the filter
chain during `PostToolFilter` or `LLMCall` states. They do not create new
states or transitions.

These two constants are added to the `EventType` vocabulary now (total: 21)
rather than deferred to Phase 4, because:

1. The `FilterDecision.Action` values (`Redact`, `Log`) in D42 need to map to
   specific event types. Phase 4's error-to-event mapping depends on these
   constants being defined.
2. Adding `EventType` constants post-freeze is technically non-breaking in Go
   (it is an additive change to a `const` block), but consumers who write
   exhaustive switch statements will miss them. Defining them now prevents a
   phase-boundary gap.

New constants:

```go
EventTypePIIRedacted                EventType = "filter.pii_redacted"
EventTypePromptInjectionSuspected   EventType = "filter.prompt_injection_suspected"
```

Phase 4 defines the semantics: when exactly these are emitted, what
`InvocationEvent` fields are populated, and how they correlate with
`FilterDecision` values. Phase 3 defines only the constant names and their
namespace.

### 52c. `budget.PriceProvider` promotion confirmed

D47 promoted `budget.PriceProvider` from `stable-v0.x-candidate` to
`frozen-v1.0`. This is already recorded in D47; this amendment note confirms it
updates the seed §5 and Phase 1 `04-v1-freeze-surface.md` tier table.

Updated tier count: 13 of 14 interfaces at `frozen-v1.0`; 1
(`identity.Signer`) at `stable-v0.x-candidate`.
