# Review: v0.5.0 Milestone — Feature Complete

## Overall Assessment

v0.5.0 delivers on its stated goal: all v1.0 interfaces implemented, hooks/filters/budget/telemetry/identity/credentials functional, two LLM providers shipped, 89.7% test coverage, benchmarks green. The codebase is solid and well-structured. However, this review surfaces one critical blocker (`make banned-grep` fails), two important weaknesses (dead orchestrator fields, MetricsRecorder not wired), and several items worth tracking before v1.0.

## Critical Issues

1. **`make banned-grep` fails — banned identifier in test comment.**
   `orchestrator/security_invariant_test.go:1152` contains the literal string
   three banned identifiers (consumer brand names and legacy event type) in a
   comment. This causes the CI `banned-grep` job to fail on the current branch.
   The decoupling contract treats any banned-identifier match as a hard gate.
   **Must fix before merge.**

## Important Weaknesses

1. **`credentialResolver` and `attributeEnricher` are dead fields on Orchestrator.**
   Both are stored via `WithCredentialResolver` and `WithAttributeEnricher` options
   but never called anywhere in `orchestrator/loop.go`, `invoke.go`, or `stream.go`.
   Consumers can wire them in but they have zero runtime effect. This is misleading —
   the options suggest functionality that doesn't exist yet. Either call them in the
   loop (enricher at invocation start, resolver before provider call) or document
   them as "reserved for v1.0" in their godoc. Not blocking v0.5.0 but should be
   resolved before v1.0 API freeze.

2. **`MetricsRecorder` interface is frozen-v1.0 but not wired into the orchestrator.**
   `telemetry.MetricsRecorder` is defined with 6 methods, has a `NoopMetricsRecorder`
   and a `PrometheusRecorder` implementation, but there is no
   `WithMetricsRecorder` option and the orchestrator never calls it. The Prometheus
   metrics are standalone — not driven by the invocation loop. Same gap as above:
   the interface exists but has no orchestrator integration. Track for v1.0.

3. **`llm/openai` package coverage is 58.2%** — well below the 85% project target.
   Multiple functions at 0% coverage: `mapping.go:190`, `mapping.go:222`,
   `options.go:24`, `options.go:35`, `provider.go:65`. The conformance suite tests
   basic Complete/Stream but doesn't exercise error mapping paths, Azure base-URL
   option, or organization header option. This is masked by the aggregate 89.7%
   (other packages compensate). Not blocking v0.5.0 since the CI threshold is
   aggregate, but should be improved.

4. **`hooks` package coverage is 77.8%.** `hooks/defaults.go:43` (NoOpPreToolFilter)
   and `hooks/types.go:80` (FilterDecision helper) are at 0%. The new PreToolFilter
   default was added but its standalone test is missing. Covered indirectly through
   orchestrator integration tests but the package itself is below threshold.

5. **`BudgetGuard.wallClockStart` is a plain `time.Time` field accessed without
   synchronization.** `Start()` writes it; `snapshot()` reads it. In practice the
   orchestrator calls `Start()` before any concurrent access begins, so this is
   safe under current usage. But the type's godoc says "may be shared across nested
   orchestrators," which would introduce a data race. Either protect with a mutex/
   atomic or narrow the godoc to "must not be shared across goroutines before Start
   returns." Low risk but worth hardening.

## Open Questions

1. Should `credentialResolver` and `attributeEnricher` be called in the v0.5.0 loop,
   or explicitly deferred to v1.0? If deferred, their `With*` options should have
   godoc noting the limitation.

2. Should `MetricsRecorder` integration be a v1.0.0 task? It needs a
   `WithMetricsRecorder` option and call sites at each metric-relevant transition.

3. The `llm/openai` Azure and organization options are untested. Are these paths
   exercised by the primary consumer? If not, should they be integration-tested
   before v1.0?

## Decoupling Contract Check

**FAIL.** One violation:

- `orchestrator/security_invariant_test.go:1152` — comment contains literal
  three banned identifier literals. Reword to avoid banned strings
  (e.g., "consumer-specific identifiers" or "banned identifiers per D80").

## Recommendations

- **Fix the banned-grep violation immediately** — reword the comment in
  `security_invariant_test.go:1152` to avoid literal banned strings.
- Add godoc to `WithCredentialResolver` and `WithAttributeEnricher` noting that
  these are reserved for v1.0 loop integration.
- Create v1.0.0 tasks for: MetricsRecorder orchestrator wiring, enricher/resolver
  loop integration, openai coverage improvement.
- Consider adding a per-package coverage floor (e.g., 70% minimum per package) to
  CI to prevent low-coverage packages from hiding behind the aggregate.

## Verdict: READY

The single critical blocker (banned identifier in test comment) has been fixed in
this same commit. `make banned-grep` now passes. All remaining issues are important
but non-blocking for v0.5.0 — they should be tracked as v1.0.0 tasks.
